# Permissions And Hooks Guide

This document extends `README.md` and focuses only on permission design for the `Projects` extension.

The goal is to explain:

- where permission decisions happen
- which hooks are emitted by `Projects`
- how consumer modules such as `UserRights` and `UserDags` use them
- how to implement new permission policies without forking the extension

## Design Principle

`Projects` is intentionally not the final permission engine.

`Projects` does three things:

1. builds list/edit/view/delete flows from the manifest
2. exposes stable runtime contexts
3. emits hooks before sensitive operations

Actual authorization is delegated to modules that subscribe to those hooks.

Today the two main consumers are:

- `milkadmin/Modules/UserRights`
- `milkadmin/Modules/UserDags`

They solve different problems and should stay separate.

## Permission Layers

There are four different permission layers.

### 1. Project/module access

Question:

- can the user enter the project at all?

Main hook:

- `projects.additional-access`

Typical consumer:

- `UserRights`

### 2. Form/table special permissions

Questions:

- can the user view this table?
- can the user edit records of this table?
- can the user soft delete / hard delete?
- can the user use extra permissions such as download?

Main hooks:

- `projects.get-special-permissions`
- `projects.check_special_permission`

Typical consumer:

- `UserRights`

### 3. Row-level visibility and access

Questions:

- which rows can the user see in the list?
- can the user open a direct edit/view URL for a specific record?
- can the user open a child form linked to a parent record they cannot access?

Main hooks:

- `projects.query.before-execute`
- `projects.data.after-extract-visibility`
- `projects.record.edit.before-render`
- `projects.record.view.before-render`

Typical consumer:

- `UserDags`
- `UserRights` when a role enables project-wide ownership filters such as `only_own_records`

### 4. Mutation control

Questions:

- can the user save this record?
- can the user create a child record under this parent?
- can the user delete / restore / hard delete this record?

Main hooks:

- `projects.record.create.before-save`
- `projects.record.update.before-save`
- `projects.record.related.before-save`
- `projects.record.delete.before-action`
- `projects.record.restore.before-action`
- `projects.record.hard-delete.before-action`

Typical consumer:

- `UserDags`
- future business-rule modules

## Hook Catalog

This section lists the hooks most relevant for permissions.

## Access Hooks

### `projects.additional-access`

Purpose:

- final decision on project entry

Used by:

- `Modules/UserRights/UserRightsModule.php`

Expected behavior:

- return `true` to allow
- return `false` to deny

Use this when the user should not even enter the project/module area.

### `projects.get-special-permissions`

Purpose:

- expose all permission keys generated by the project manifest

Used by:

- `Extensions/Projects/Module.php`
- `Modules/UserRights/UserRightsPermissionTrait.php`

This is not an access decision hook. It is a discovery hook used by `UserRights` to build the role UI and stored permission structure.

### `projects.check_special_permission`

Purpose:

- runtime check for table/form permission keys such as:
  - `project.<manifest_id>.main_table_view`
  - `project.<manifest_id>.main_table_edit`
  - `project.<manifest_id>.<child_form>__view`
  - `project.<manifest_id>.<child_form>__edit`
  - `project.<manifest_id>.download`

Used by:

- `Projects`
- `UserRights`

Expected behavior:

- first arg is the current decision, usually `true`
- consumer can return `false` to deny

This is the right hook for role permissions. It is not the right hook for row filtering.

## List Hooks

### `projects.query.before-execute`

Purpose:

- mutate the list query before execution

Used by:

- `UserDags`

Emitted for:

- rows query
- total/count query

Important payload fields:

- `query`
- `is_total`
- `query_type`
- `page`
- `table_id`
- `context`
- `manifest`
- `model`
- `model_columns`
- `request`

Typical use:

- add `where`
- add `whereIn`
- add joins or extra filters

This is the main hook for row-level restriction on root tables.

### `projects.data.after-extract-visibility`

Purpose:

- evaluate extracted rows after the query

Used by:

- `UserDags`

Typical use:

- correlated tables where visibility cannot be enforced with a single direct `dag_id` clause
- block the page if rows belong to a parent/root record the user cannot access

This hook should be treated as a second line of defense, not as the only filter when a SQL filter is possible.

## Render Hooks

### `projects.record.edit.before-render`

Purpose:

- block direct access to edit pages before the form is shown

Used by:

- `UserDags`

Important cases:

- root record edit
- child record edit
- new child record creation page without `id`

For child create pages, there may be no current record yet. In that case the consumer must resolve the permission from the parent/root context.

### `projects.record.view.before-render`

Purpose:

- block direct access to view pages before rendering

Used by:

- `UserDags`

This protects direct URLs even when the row is already hidden in the list.

## Save Hooks

### `projects.record.create.before-save`

Purpose:

- inspect or mutate request data before creating a new record

Typical use:

- assign default DAG
- enforce ownership fields
- block create based on project policy

### `projects.record.update.before-save`

Purpose:

- inspect or mutate request data before updating an existing record

Typical use:

- prevent unauthorized updates
- keep immutable fields unchanged

### `projects.record.related.before-save`

Purpose:

- extra hook for child/non-root forms

Typical use:

- validate that the child record belongs to a root record the user can manage

## Delete Hooks

### `projects.record.delete.before-action`

Purpose:

- block delete before soft delete or hard delete path starts

Emitted from:

- list delete flow
- delete page flow
- delete confirm flow

Typical use:

- deny delete when record DAG is not accessible

### `projects.record.restore.before-action`

Purpose:

- block restore of soft-deleted rows

### `projects.record.hard-delete.before-action`

Purpose:

- block permanent delete

## How UserRights Works

`UserRights` is role-based.

It answers:

- can the user access this project?
- which tables/actions are allowed in this project?

Main responsibilities:

### 1. Project entry

`UserRights` subscribes to:

- `projects.additional-access`

If the user has no assigned role for the project, access is denied.

### 2. Manifest-driven permission discovery

`Projects` publishes special permissions through:

- `projects.get-special-permissions`

`UserRights` reads them and builds the permission UI dynamically.

This avoids hardcoding the list of permission keys inside `UserRights`.

### 3. Runtime permission check

`Projects` asks:

- `Hooks::run('projects.check_special_permission', true, 'project.' . $permissionName)`

`UserRights` answers by checking the current user role JSON.

This covers:

- view
- edit
- soft delete
- hard delete
- additional manifest permissions

### 4. Module visibility

`UserRights` also alters project/module visibility through:

- `projects.rule-apply-from-manifest`

This is used to hide menu entries when the user has no project role.

### 5. Project-wide data policies

`UserRights` can also apply project-wide role policies that are not tied to a single table permission.

Current example:

- `only_own_records`

When enabled for the current user's role, `UserRights` subscribes to:

- `projects.query.before-execute`
- `projects.record.edit.before-render`
- `projects.record.view.before-render`
- `projects.record.create.before-save`
- `projects.record.update.before-save`
- `projects.record.related.before-save`
- `projects.record.delete.before-action`
- `projects.record.restore.before-action`
- `projects.record.hard-delete.before-action`

and enforces:

- `created_by = current_user_id` on list queries when the current table exposes `created_by`
- direct edit/view denial when the current record is not owned by the user
- create/update denial when the record or main/root record is not owned by the user
- delete/restore/hard-delete denial when the record or main/root record is not owned by the user

This is a project-wide ownership filter and is intentionally separate from DAG logic.

## How UserDags Works

`UserDags` is data-partition based.

It answers:

- among the records of a project, which subset can this user access?

Main responsibilities:

### 1. List filtering

Hook:

- `projects.query.before-execute`

Behavior:

- if the user has DAG assignments for the project, the query is filtered to allowed DAGs
- if the user has no DAG assignments, `UserDags` does not restrict the query
- admin bypasses the restriction

This is the first and most efficient filter.

### 2. Correlated-table visibility

Hook:

- `projects.data.after-extract-visibility`

Behavior:

- for tables that do not expose `dag_id` directly, `UserDags` resolves the root record DAG after extraction
- if rows are not safely visible, the page is blocked

### 3. Direct edit/view protection

Hooks:

- `projects.record.edit.before-render`
- `projects.record.view.before-render`

Behavior:

- root edit/view is checked against the record DAG
- child edit/view is checked against the root DAG
- new child form pages are checked against the parent/root DAG even if there is no current child record yet

This closes the common bypass where a user guesses a direct URL.

### 4. Save protection

Hooks:

- `projects.record.create.before-save`
- `projects.record.update.before-save`
- `projects.record.related.before-save`

Behavior:

- on create, if the user has a primary DAG, it can be assigned automatically to the new root record
- on update, the user must belong to the same DAG as the main/root record
- `dag_id` is not meant to be changed arbitrarily in edit
- child records inherit authorization from the root record DAG

### 5. Delete/restore protection

Hooks:

- `projects.record.delete.before-action`
- `projects.record.restore.before-action`
- `projects.record.hard-delete.before-action`

Behavior:

- all mutations are denied if the target record/root record belongs to an inaccessible DAG

## Recommended Development Pattern

When implementing a new permission module, do not put all logic into one hook.

Use this pattern:

1. project access via `projects.additional-access`
2. table/form capabilities via `projects.check_special_permission`
3. list restriction via `projects.query.before-execute`
4. post-query verification via `projects.data.after-extract-visibility` only when SQL filtering is insufficient
5. direct URL protection via `projects.record.*.before-render`
6. mutation enforcement via `projects.record.*.before-save` and `projects.record.*.before-action`

This keeps responsibilities separate and easier to debug.

## Recommended Payload Contract For Consumers

If your hook mutates data, prefer these conventions.

### Query hook

Modify:

- `payload['query']`

Do not rely on:

- `sql`
- `params`

Those fields are diagnostic snapshots, not the source of truth.

### Save hooks

Modify:

- `payload['request']`

Deny save by:

- adding an error message
- returning the payload with the unchanged request when your module already stops the process downstream

### Delete hooks

Return:

- `payload['allowed'] = false`
- `payload['message'] = '...'`

This keeps the control flow explicit and avoids hard exits inside the consumer.

## What Belongs In UserRights Vs UserDags

Use `UserRights` for:

- role permissions
- form/table action toggles
- project entry permissions
- permissions that apply independently from the current record values

Use `UserDags` for:

- row partitioning
- root/child visibility based on dataset ownership groups
- mutation rules derived from DAG membership

Do not put row partition logic into `UserRights`.

Do not put role matrix logic into `UserDags`.

They can be combined, but they solve different axes of authorization.

## Suggested Future Permission Axis: "Only Own Records"

This is a separate concern from DAG.

`DAG` means:

- which data partition/group a user can access

"Only own records" means:

- within the allowed partition, the user can see or edit only records created by them, assigned to them, or linked to a chosen owner field

Architecturally this belongs more to `UserRights` than to `UserDags`.

Reason:

- it is a policy toggle
- it is usually role-driven
- it may vary by form
- it is orthogonal to DAG

Recommended model:

- keep DAG filtering in `UserDags`
- add an extra permission mode in `UserRights`, for example:
  - all visible records
  - own visible records only

Runtime combination should be:

- `final access = role policy AND DAG policy`

So if a user has access to DAG 1 and role policy says "own only", they should only access:

- records in DAG 1
- that also match the ownership rule

## Practical Advice For New Hook Consumers

Before adding a new hook consumer:

1. decide if the rule is role-based, row-based, or both
2. choose the lowest hook that can enforce it correctly
3. if SQL filtering is possible, do it in `projects.query.before-execute`
4. always protect direct URLs with render hooks
5. always protect save/delete with mutation hooks

If you skip steps 4 and 5, the list may look correct but direct URLs will still bypass the policy.
