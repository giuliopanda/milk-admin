# Projects Extension JSON Options

This document lists the JSON options supported by the `Projects` extension.

There are four JSON types:

1. `Project/manifest.json` (project structure, nested forms).
2. `Project/<FormName>.json` (form schema that extends a model RuleBuilder).
3. `Project/view_layout.json` (layout definition for root `*-view` pages when single-record view is enabled).
4. `Project/search_filters.json` (search filters definition for root `*-list` pages).

Everything below is based on the current implementation in:

- `milkadmin/Extensions/Projects/Classes/ProjectManifestParser.php`
- `milkadmin/Extensions/Projects/Classes/ModelJsonParser.php`
- `milkadmin/Extensions/Projects/Classes/SchemaJsonAdapter.php`

For full `view_layout.json` reference, see:

- `milkadmin/Extensions/Projects/VIEW_LAYOUT.md`

## `Project/search_filters.json` (Root List Search Filters)

This file is optional.

When present, it configures:

- Search/filter UI rendering (via `SearchBuilder`) for root `*-list` pages.
- SQL filter callbacks bound to the root `TableBuilder`.

Important runtime rule:

- Filters are applied only to the main/root list table.
- Child/subtable lists are never affected.

Supported shapes:

1. Single/default config:

```json
{
  "search_mode": "submit",
  "url_params": {
    "user_id": {
      "type": "int",
      "field": "user_id",
      "operator": "equals",
      "required": true
    }
  },
  "filters": [
    {
      "type": "search",
      "name": "search",
      "label": "Search",
      "field": "title",
      "operator": "like"
    }
  ]
}
```

2. Multi-form config:

```json
{
  "forms": {
    "RootFormA": { "filters": [ ... ] },
    "RootFormB": { "filters": [ ... ] },
    "*": { "filters": [ ... ] }
  }
}
```

Main keys:

- `search_mode` (`submit`, optional, default `submit`)
- `auto_buttons` (bool, optional, default `true`)
- `wrapper_class`, `form_classes`, `container_classes` (string, optional)
- `url_params` (object or array, optional)
- `filters` (array, optional)

`url_params` declares **whitelisted URL query params** accepted by the form.
Each accepted param can be:

- object shape:
  - `"url_params": { "user_id": { ...options... } }`
- array shape:
  - `"url_params": [ { "name": "user_id", ...options... } ]`

Supported `url_params` options:

- `name` (required only in array shape)
- `field` (SQL field, optional, default = param name)
- `operator` (same operators as filter query, default `equals`)
- `type` (`int` | `float` | `bool` | `string` | `uuid` | `slug`, default `string`)
- `required` (bool, optional, default `false`)
- `max_length` (int, optional, default depends on type)

Runtime behavior:

- Param is sanitized by declared `type`.
- If valid, it is applied as SQL WHERE on root list.
- If valid, it is added as `customData(param, value)` on `TableBuilder` and `FormBuilder`.
- Param is propagated in row action URLs (`*-edit`, `*-view`) to keep context.
- If `required=true` and missing/invalid, list returns no rows (fail-closed).

Filter entry keys:

- `type` (`search` | `select` | `action_list` | `input` | `search_button` | `clear_button`)
- `name` (required for non-button types)
- `label`, `placeholder`, `class`, `layout` (optional)
- `input_type` (for `input`, optional)
- `options` (for `select` / `action_list`, optional)
- `default` (optional)
- query keys:
  - `field` (string) or `fields` (array of strings)
  - `operator` (`like` | `equals` | `starts_with` | `ends_with` | `greater_than` | `greater_or_equal` | `less_than` | `less_or_equal` | `between`)
  - aliases accepted:
    - `>` / `gt` => `greater_than`
    - `>=` / `gte` => `greater_or_equal`
    - `<` / `lt` => `less_than`
    - `<=` / `lte` => `less_or_equal`

Notes:

- If query fields are missing, the parser falls back to `name` when possible.
- A `search_filters` object can also be embedded directly in `manifest.json` root;
  file `Project/search_filters.json` has priority when both are present.
- For `between`, the filter input must contain both bounds in a single value.
  Supported formats:
  - `10,20`
  - `10..20`
  - `10 - 20`

## 1) `Project/manifest.json`

### Root Keys

- `_version` (string, optional)
- `_name` (string, optional)
- `settings` (object, optional)
  - Any keys are allowed. The extension currently does not enforce specific settings.
- `menu` (string, optional)
  - Overrides the module sidebar label generated by PHP `->menu(...)`.
- `menuIcon` (string, optional)
  - Overrides the module sidebar icon class (example: `"bi bi-folder2-open"`).
- `selectMenu` (string|object, optional)
  - Enables module grouping via `ModuleRuleBuilder::selectMenu(...)`.
  - String form:
    - `"selectMenu": "Tests"` (group name only).
  - Object form:
    - `name` / `menu` / `group` (group name, required),
    - `label` or `title` (submenu label, optional),
    - `url` (submenu URL, optional),
    - `icon` (submenu icon, optional),
    - `order` (submenu order, optional).
  - Aliases accepted: `selectedMenu`, `select_menu`.
- `forms` (array, required)
  - A list of form nodes (root forms).

### Form Node Keys (recursive)

Each node in `forms` (and each nested `forms` child) supports:

- `ref` (string, required)
  - Schema filename, relative to the module `Project/` folder.
  - Example: `"ref": "VisitBaseline.json"`.
- `max_records` (int|string, optional)
  - Allowed values:
    - `1` or `"1"`: single record per parent (hasOne-like behavior)
    - integer `>= 2` (or numeric string): finite max records per parent
    - `"n"`: multiple records per parent (hasMany-like behavior)
    - `"unlimited"`: treated like `"n"` (reserved for future)
  - Default: `"n"`.
  - Runtime navigation note:
    - for `max_records: 1` without nested child forms, `*-list` redirects to `*-edit`;
    - for `max_records: 1` with nested child forms, `*-list` stays available (max one row) to keep child navigation accessible.
- `showIf` (string, optional)
  - ExpressionParser condition evaluated against the immediate parent record.
  - If false, child navigation and child list/edit routes are blocked.
- `showIfMessage` (string, optional)
  - Custom message shown when `showIf` is false.
  - Used both in blocked child icon tooltip and child list/edit access error page.
- `viewAction` (bool|int|string, optional)
  - If enabled, registers an additional `*-view` action for the form.
  - Accepted truthy values: `true`, `1`, `"1"`, `"true"`, `"yes"`, `"on"`.
  - Alias accepted: `view_action`.
- `childCountColumn` (string, optional)
  - Controls visibility of direct-child count columns in the form list.
  - Allowed values:
    - `"hide"`: always hide count columns
    - `"show"`: always show count columns
    - omitted/invalid: default behavior
  - Default behavior:
    - when `viewSingleRecord/viewAction` is active on root list, count columns are hidden;
    - otherwise count columns are shown.
  - Alias accepted: `child_count_column`.
- `viewDisplay` (string, optional)
  - How `*-view` is opened for this form.
  - Allowed values: `"page"` (default), `"offcanvas"`, `"modal"`.
  - Alias accepted: `view_display`.
- `listDisplay` (string, optional)
  - How `*-list` is opened for this form.
  - Allowed values: `"page"` (default), `"offcanvas"`, `"modal"`.
  - Alias accepted: `list_display`.
- `editDisplay` (string, optional)
  - How `*-edit` is opened for this form.
  - Allowed values: `"page"` (default), `"offcanvas"`, `"modal"`.
  - Alias accepted: `edit_display`.
- Note:
  - When opened without fetch, actions still render as normal pages.
  - When opened in fetch mode, generated links/buttons automatically include fetch attributes.
  - Edit forms opened in fetch mode automatically reload the correct source table after save.
- `forms` (array, optional)
  - Children form nodes (same structure).
- `existingTable` (bool|int|string, optional)
  - Marks the form as linked to an already existing DB table.
  - Intended for cases where table structure is externally managed and must stay read-only in Projects build forms page.
  - Accepted truthy values: `true`, `1`, `"1"`, `"true"`, `"yes"`, `"on"`.
  - Alias accepted: `existing_table`.

### Legacy Keys

Legacy manifest selector keys are **not supported** in the current parser:

- `settings.default_form`
- `settings.default_form_ref`
- `default_form`

## 2) `Project/<FormName>.json` (Schema JSON)

This is the schema used to extend the model RuleBuilder.

Important:

- The extension currently applies only the `model` section.
- Keys starting with `_` are treated as metadata and ignored by the model parser (but may be preserved by `SchemaJsonAdapter`).

### Metadata Keys (root-level)

These are recognized as metadata by `SchemaJsonAdapter`:

- `_version`
- `_name`
- `_description`
- `_author`
- `_created`
- `_updated`

Additionally, any key that starts with `_` is treated as metadata (ignored by section handlers).

### Sections (root-level)

Currently supported/used by this extension:

- `model` (object)

Any other root-level section is ignored by the Projects extension (but can be preserved by `SchemaJsonAdapter` as `_unhandled`).

## 2.1) `model` Section Keys

Supported keys in `"model": { ... }`:

- `table` (string, optional)
  - If provided, calls RuleBuilder `->table(...)`.
- `db` (string, optional)
  - If provided, calls RuleBuilder `->db(...)`.
- `extensions` (array, optional)
  - Passed to RuleBuilder `->extensions([...])`.
- `rename_fields` (object map, optional)
  - `{ "old_name": "new_name" }`
  - Each entry calls `->renameField($from, $to)`.
- `removePrimaryKeys` (bool, optional)
  - If `true`, calls `->removePrimaryKeys()`.
- `fields` (array, optional)
  - List of field definitions (see below).

## 2.2) Field Definition Keys

Each entry in `model.fields[]` supports:

### Required Keys

- `name` (string, required)

### Creation / Replacement

- `method` (string, optional)
  - Default: `"string"`.
  - Supported values are listed in `ModelJsonParser::VALID_METHODS`:
    - `id`, `primaryKey`
    - `string`, `title`, `text`
    - `int`, `decimal`
    - `datetime`, `date`, `time`, `timestamp`, `created_at`
    - `email`, `tel`, `url`
    - `file`, `image`
    - `boolean`, `checkbox`
    - `checkboxes`, `radio`
    - `list`, `select`, `enum`
    - `array`
    - `field` (generic field definition)
  - Note (`method` vs RuleBuilder `type`):
    - `method` belongs to the Projects JSON DSL (external schema format).
    - RuleBuilder internally stores field kind as `type` (runtime format).
    - Parser maps JSON `method` to RuleBuilder calls (example: `method: "decimal"` -> `->decimal(...)`, internal `type: "float"`).
    - Serializer maps internal rules back to JSON `method`.
- `replace` (bool|int|string, optional)
  - If `true` / `1` / `"1"`, forces re-creation of the field when it already exists in the current RuleBuilder.
  - Fields that already exist in the **PHP model RuleBuilder** are always ignored by schema JSON (regardless of `replace`).

### Type Parameters (used by some methods)

- `type` (string, optional)
  - Used when `method` is `field` (generic field).
- `length` (int, optional)
  - Used by `string`, `title`, `decimal`.
- `precision` (int, optional)
  - Used by `decimal`.
- `dbType` (string, optional)
  - Used when `method` is `text` to pick MySQL text variant.
  - Allowed values: `tinytext`, `text`, `mediumtext`, `longtext`.
- `options` (object|array|string, optional)
  - Used by `checkboxes`, `radio`, `list`/`select`, `enum`.
  - Can be:
    - an object/array (`{ "k": "Label" }`)
    - a callable-like string (example: `"\\StaticData::Days()"`) resolved via callable resolver.

### Standard Config Keys (mapped to RuleBuilder methods)

These keys map directly to RuleBuilder field configuration methods:

- `label`
- `default`
- `formType`
- `formLabel`
- `error`
- `calcExpr`
- `step`
- `min`
- `max`
- `accept`
- `uploadDir`
- `saveValue`
- `requireIf`
- `maxFiles`
- `maxSize`
- `formParams`
- `properties`

### Flag Config Keys (boolean)

If the key is present and `true`/`1`/`"1"`, it calls the corresponding RuleBuilder method:

- `required`
- `unique`
- `index`
- `hideFromList`
- `hideFromEdit`
- `hideFromView`
- `hide`
- `excludeFromDatabase`
- `unsigned`
- `noTimezoneConversion`

### Special Handling Keys

- `nullable` (bool)
  - Calls `->nullable(true|false)`.
- `builderLocked` (bool|int|string)
  - Locks the field in the Projects table editor page (`build-form-fields`).
  - When enabled, the field is treated as intoccabile and is not shown in the "Campi del model (RuleBuilder)" list.
  - Accepted truthy values: `true`, `1`, `"1"`, `"true"`, `"yes"`, `"on"`.
  - Alias accepted: `builder_locked`.
  - This key is editor metadata (it does not map to a RuleBuilder method).
- `multiple` (bool|int)
  - Calls `->multiple(...)`.
- `validateExpr` (string|array)
  - If string: `->validateExpr($expr)`
  - If array: `->validateExpr($expr[0], $expr[1] ?? null)`
- `apiUrl` (string|object)
  - If string: `->apiUrl($url, $apiDisplayField ?? null)`
  - If object: `->apiUrl($apiUrl.url, $apiUrl.display_field ?? null)`
- `apiDisplayField` (string)
  - Used only when `apiUrl` is provided as a string.
- `checkboxValues` (array)
  - `->checkboxValues($checked, $unchecked)`
- `property` (array)
  - `->property($property[0], $property[1])` (requires at least 2 elements).

## 2.3) Relationship Keys

Relationships are defined inside a field definition.

### `belongsTo`

```json
{
  "name": "teacher_id",
  "method": "int",
  "belongsTo": {
    "alias": "teacher",
    "related_model": "SociModel",
    "related_key": "id",
    "where": { "condition": "active = ?", "params": [1] }
  }
}
```

Supported keys:

- `alias` (string, required)
- `related_model` (string, required)
- `related_key` (string, optional, default `"id"`)
- `where` (object, optional)
  - `condition` (string)
  - `params` (array)

### `hasOne`

```json
{
  "name": "id",
  "method": "id",
  "hasOne": {
    "alias": "profile",
    "related_model": "ProfileModel",
    "foreign_key": "user_id",
    "onDelete": "CASCADE",
    "allowCascadeSave": false
  }
}
```

Supported keys:

- `alias` (string, required)
- `related_model` (string, required)
- `foreign_key` (string, required)
- `onDelete` (string, optional, default `"CASCADE"`)
- `allowCascadeSave` (bool, optional, default `false`)
- `where` (object, optional) same shape as above

### `hasMany`

Same keys and structure as `hasOne`.

### `withCount`

```json
{
  "name": "id",
  "method": "id",
  "withCount": {
    "alias": "comments_count",
    "related_model": "CommentsModel",
    "foreign_key": "post_id"
  }
}
```

You can also provide an array of `withCount` objects:

```json
{ "withCount": [ { ... }, { ... } ] }
```

Supported keys:

- `alias` (string, required)
- `related_model` (string, required)
- `foreign_key` (string, required)
- `where` (object, optional) same shape as above

## Runtime FK Conventions (manifest-driven)

Beyond JSON options, the Projects extension injects FK fields at runtime for child forms:

- Immediate parent FK: `<parent_form_snake>_id`
- Root closure FK: `root_id`

`root_id` always contains the root record id of the current manifest branch, also for first-level children (where it equals the immediate parent id).

When a new child form schema is auto-generated by `Modules/Projects`, these FK fields are pre-created in `model.fields[]` with:

- `required: true`
- `hideFromList: true`
- `formParams: { "readonly": true }`
- `builderLocked: true`

Additionally, when a new child model PHP is auto-generated, the same FK fields are written directly in `configure()` (`<parent>_id` and `root_id`) as required + readonly + hidden from list.

Build editor delete behavior:

- Fields defined directly in model PHP `configure()` are visible but cannot be deleted from build editor.
- Fields not defined in model PHP can be deleted only if they are not `builderLocked`.
- The detection of "defined in model PHP" is performed without extensions via `AbstractModel::getRulesDefinedInModelConfigureOnly()`.
- Child FK technical fields (`<parent>_id`, `root_id`) are always hidden from the build editor list (intoccabili), even if `builderLocked` is removed manually from JSON.

Build project edit UI (module-level note):

- Project editing pages use a shared context box that shows current section and edited project.
- Main project edit pages (`edit`, `build-forms`, `build-form-fields`) are visually aligned to the left using shared layout classes.
- Back navigation for project edit pages is centralized in the shared section header (same button style/position across pages).

### `hasMeta`

`hasMeta` can be a single object or an array of objects:

```json
{
  "name": "id",
  "method": "id",
  "hasMeta": {
    "alias": "meta",
    "related_model": "MetaModel",
    "foreign_key": "post_id",
    "local_key": "id",
    "meta_key_column": "meta_key",
    "meta_value_column": "meta_value",
    "meta_key_value": "category"
  }
}
```

Supported keys:

- `alias` (string, required)
- `related_model` (string, required)
- `foreign_key` (string, optional)
- `local_key` (string, optional)
- `meta_key_column` (string, optional, default `"meta_key"`)
- `meta_value_column` (string, optional, default `"meta_value"`)
- `meta_key_value` (string, optional)
- `where` (object, optional) same shape as above

## 2.4) `list_options` (Field List Display Options)

When a field has `visibility.list = true`, the draft field config can include a `list_options` object to control how the column is rendered in the list table.

The `list_options` object is optional. When omitted, the column renders with default behavior (plain text, no transformation).

All sub-keys are optional and independent of each other.

```json
{
  "name": "status",
  "method": "string",
  "list_options": {
    "link": {
      "url": "?page=orders&action=view&id=%id%",
      "target": "same_window"
    },
    "html": true,
    "truncate": 50,
    "change_values": {
      "1": "Active",
      "0": "Inactive",
      "draft": "Draft"
    }
  }
}
```

### `list_options.link`

Renders the column value as a clickable link.

- `url` (string, required) - Link URL. Supports `%field_name%` placeholders replaced with row values at render time. Example: `?page=orders&id=%id%&type=%status%`.
- `target` (string, optional, default `"same_window"`) - `"same_window"` or `"new_window"`.

### `list_options.html`

- (bool, default `false`) - When `true`, the column value is rendered as raw HTML instead of escaped text.

### `list_options.truncate`

- (int, default `50` when enabled) - Maximum number of characters to display. Text exceeding this length is truncated with ellipsis.

### `list_options.change_values`

Maps stored values to display labels. The column shows the mapped label instead of the raw stored value.

- (object) - Key-value pairs where key is the stored value and value is the display label.

```json
{
  "change_values": {
    "1": "Yes",
    "0": "No",
    "pending": "Pending Review",
    "approved": "Approved"
  }
}
```

The editor UI provides the same Manual/Text interface used by select field options:

- **Manual tab**: sortable list with value/label inputs, drag-to-reorder.
- **Text tab**: bulk editing with one mapping per line, format `value|display label`.

## Debug: Detect Ignored Fields

When applying a schema to an existing model RuleBuilder, any `model.fields[]` entry whose `name` already exists in the PHP model rules is ignored.

You can preview the ignored fields list via:

- `Extensions\\Projects\\Classes\\ModelJsonParser::analyzeIgnoredFields($modelSection, $ruleBuilder)`
- `Extensions\\Projects\\Classes\\ModelSchemaSection::analyzeIgnoredFields($modelSection, $ruleBuilder)`

## Build Form Fields Save/Review Flow

The `build-form-fields` editor now uses a draft + review step for schema JSON writes:

1. Click `Save` in `build-form-fields` (no immediate write to file).
2. A review page shows:
   - the current JSON file content
   - the new JSON draft content (as it would be saved)
3. Click `Accept changes` to overwrite the target `Project/<Form>.json`.
4. After accept, redirect goes to:
   - `?page=projects&action=build-forms&module=<ModuleName>`
5. If changes are not accepted:
   - `Back to editor without saving` returns to `build-form-fields`
   - the editor reloads the unsaved draft data (same form/module).
